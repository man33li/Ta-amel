# v2.0 Roadmap ÔÇö AI Memory + Spatial UX

## Phase 1 ÔÇö schema + memory backbone

**Status:** Ôťô shipped

- Migration `supabase/migrations/0002_palace_schema.sql` adds `wings`, `rooms`, FK on `cards`, pgvector extension, FTS index on cards, RLS policies.
- `mem0ai` added as a runtime dependency. `OPENAI_API_KEY` and `SUPABASE_DB_URL` documented in `.env.example`.
- `src/lib/memory/client.ts` exposes a typed `MemoryAdapter` interface with `add / search / remove`. Singleton, server-only, gracefully no-ops when env vars missing.
- `src/app/api/memory/sync/route.ts` and `src/app/api/memory/search/route.ts` wrap the adapter; `runtime = 'nodejs'`.
- `src/lib/memory/merge.ts` merges semantic + keyword + recency lanes via RRF (k=60).
- `useNotes.updateNote` fires `/api/memory/sync` after each successful update; failure is swallowed.
- Tests: 6 RRF merge cases.

## Decisions

| Decision | Rationale |
|---|---|
| Adopt `mem0ai/oss` instead of cloud Mem0 | Keeps data in our existing Postgres; one API key (OpenAI) instead of two |
| PGVector on Supabase, not a separate vector DB | No new infra; pgvector is already available |
| Fire-and-forget memory sync, not blocking | Editor must stay snappy; semantic recall is best-effort |
| RRF over weight tuning | mempalace's hybrid v4 idea ÔÇö robust without per-query tuning |
| Wings/rooms are ephemeral selections | URL-less spatial nav matches a "walking through the palace" feel |
| Move control opt-in via props on NoteCard | Doesn't break existing screens; palace renders it where useful |
| No knowledge graph in v2.0 | Earn it with usage data first |
